Prelude

If you struggle learning Haskell as a Pythonista or C++ nerd, or even a ninja who have strong background in other imperative languages, you're definitely not the only one. Functional programming is a different paradigm which can apply to other modern languages like Ruby, Python, JS, etc. But learning to write pure functional code in Haskell is totally a different topic. Developers with a strong mindset trained for imperative or procedural programmings may find it noticeably hard to adopt different thinking for Haskell. Knowing lots of imperative languages doesn't seem to take advantage.
ALERT: The article is based on personal experience. YMMV.

Doom of parentheses

You may want to have a look at a complete list of Haskell block syntaxes if you haven't realized what they are. In general, Haskell encourages the programmers to write redundant parentheses less. Most of Haskell syntaxes don't even need parentheses to group such expressions. This ultimately makes the code clean and minimized.
Let's take a look at the following code. 
				
					u = realToFrac( head( zip ((*) a b ) ) )

This is simply not looking good in Haskell. You can remove unnecessary parentheses while the resultant code is still sound and legal as follows. 

				
					u = realToFrac $ head $ zip (*) a b

Gorgeous, neat and very nice looking. Let's then try another example which is a bit more complicated. 

				
					-- Say we have this at first
					take(3 reverse( zipWith( (\a b -> (a-b)/(a+b)) [599,598..1] [290,280..0] ) ))

					-- Remove unnessary parentheses
					-- Group tokens with $ sign
					take $ 3 reverse $ zipWith $ (\a b -> (a-b)/(a+b) [599,598..1] [290,280..]

However, in reality for beginners, the boundary of grouping could be unbelievably confusing ... 

				
					-- This breaks! 
					take 3 $ reverse $ zipWith $ (\a b -> (a-b)/(a+b) [599,598..1] [290,280..]

					-- This also does!
					take $ 3 reverse $ zipWith (\a b -> (a-b)/(a+b) [599,598..1] [290,280..]

Can you spot the mistakes and give reasons why they are disfunctional?

Indentations hurt

Pythonistas might not feel uncomfortable with the complexity of how identations make such difference in terms of meanings in the code. Improper identations cause syntactic errors which most of the time the error messages are slightly misleading. In Python, the statements are indented to make the code structure well-organized in hierachy. In Haskell, identation causes statements to nest.

The following code snippet demonstrates a general indentation case. 

				
					countEql a b = 
					  let ab = [a'==b' | (a',b') <- zip a b]
					  in length $ filter (\n -> n) ab
But when it comes into binding do, the same indentation will break: 
				
					something $ do
					  let ab = [a'==b' | (a',b') <- zip a b]
					  in bindValue $ length $ filter (\n -> n) ab -- BREAKS
Surprisingly, indent the `in` by several spaces will fix it. 
				
					something $ do
					  let ab = [a'==b' | (a',b') <- zip a b]
					    in bindValue $ length $ filter (\n -> n) ab -- works!
My celebrum hurts ...

Still more like this awaiting to find. The language is beautiful but, still, I don't think it seems pretty friendly for new learners. It is somewhat like a smart quiet guy who is difficult to understand at first, but once you get along with him well as time goes, he will become a great fellowship.

Passing negative number

You may do this when you want to pass a negative number to a function in Python. 
				
					def f(a,b):
						return a*b

					f(5,3) # Just works
					f(-5,3) # Just works
But it hurts you when you try to do the same thing in Haskell. 
				
					let f a b = a*b

					f 5 3 -- works
					f -5 3 -- WTF! Non type-variable argument in the constraint: Num (a -> a -> a)
Haskell sees -5 as an operation "minus something by five", not a number. This would blow your mind. Function first class doesn't always magically make life easier, it somehow hurts at the first glance.

To cope with this, just wrap around it with parentheses. 

				
					let f a b = a*b

					f 5 3 -- works
					f (-5) 3 -- Awesome

Left arrows, right arrows ...

In Haskell universe, <- and -> are utilized in various areas to serve different purposes. For newbies, this is visually a nightmare. 
				
					[a | a <- [1..n]]
					..
					..
					[(a,b) | a <- [1..n], b <- [1,n..2*n]]
					..
					..
					case u of
					  Vector i,j -> (i,j)
					  Scalar c -> (0,0)
					..
					..
					map (\n -> n^2) [1..]
					..
					..
When they co-exist in a statement, somehow it hurts the beginners. 
				
					case u of
					  Nothing -> 0
					  Just v -> map (\i -> if i then [1..5] else []) [n>k| n<-v, k<-[1..3]]
Beginners may probably find it painful to start googling for the explanation. It's difficult to say what the are, how to tokenize them into expressions and how they are called. Thus, they don't even know how to google (like I was). But, when you thrive learning the syntax, you'll eventually understand how it works very quickly.
Learning Haskell is great. The only caveat is it'll be hard to tackle the tough learning curve unless you remove all senses trained specifically for OOP or imperative languages.
